# GeoDoc AI 0.4.0 — full project analysis and professional redesign report

## Что было не так

### UI / UX
- Интерфейс выглядел как перегруженный AI-dashboard: action rail, evidence panel, canvas, copilot panel, summary bar и tabs одновременно конкурировали за внимание.
- Правый copilot был слишком узким, из-за чего ответ обрезался и выглядел как сломанная карточка.
- Ответ дублировался в разных местах: canvas, copilot, история. Это создавало ощущение шума и непрофессиональности.
- На карте слишком много декоративных элементов вокруг изображения, но мало полезного смысла.
- Много терминов на английском внутри русскоязычного продукта: Evidence Index, Copilot, Workspace, etc.
- Не было нормального hierarchy: пользователь не понимал, где главный ответ, где источники, где таблицы.
- Сайт был визуально тяжёлым: много границ, капсул, маленького текста, переусложнённых карточек.
- Mobile/tablet сценарий был хрупким: основной input мог теряться или становиться неудобным.

### Ответы AI-агента
- Ответы начинались с жёсткого `ОТВЕТ`, выглядели машинно и не как продуктовый assistant.
- Map answer мог выводить плохой технический title вроде: «карта по визуальным признакам контурной/структурной карты, даже если vision-модель...».
- В ответ попадали внутренние объяснения классификатора вместо нормального геологического вывода.
- Confidence и ограничения были, но не объяснялись человеку понятным языком.
- Для карты не было стабильной структуры: краткий вывод → что видно → источник → ограничения.
- Qwen-ответ мог быть слишком длинным или слишком общим, UI не защищал пользователя от плохого формата.

### Frontend architecture
- `frontend/src/main.tsx` был большим монолитом примерно на 1500 строк.
- `frontend/src/styles.css` был более 1000 строк и обслуживал слишком сложный layout.
- UI содержал множество disabled/fake affordance-элементов.
- CSS был перегружен декоративными эффектами и нестабильными panel/layout решениями.

### Backend / RAG / AI pipeline
- Базовая безопасность уже была улучшена в 0.3.1, но response quality всё ещё зависел от качества raw title/metadata.
- Нужен был post-processing LLM-ответа, чтобы Qwen не ломал формат продукта.
- Нужна была очистка visual titles и visual text перед выдачей пользователю.
- Map response должен быть deterministic и evidence-first, а не просто «что сказала модель».

## Что исправлено в 0.4.0

### Новый минималистичный frontend
- Полностью переписан `frontend/src/main.tsx`.
- Полностью переписан `frontend/src/styles.css`.
- Убран action rail и тяжёлый dashboard-подход.
- Новый layout: левый sidebar для загрузки/индекса, центральный main для ответа и артефактов, правый evidence rail.
- Главный ответ теперь один, читаемый, широкий и не обрезается.
- Вкладки упрощены до 4 понятных разделов:
  - Ответ
  - Таблицы
  - Карты и рисунки
  - Источники
- Composer всегда доступен внизу main area, а на mobile фиксируется снизу экрана.
- Убраны неработающие кнопки «Экспорт», «Сравнить», «Full screen» и т.п.
- Снижена визуальная шумность: меньше капсул, меньше декоративных линий, больше воздуха.
- Улучшена читаемость markdown, списков и таблиц.
- Сборка стала легче: CSS уменьшен примерно с 30.5 kB до 12.7 kB, JS — с ~192 kB до ~163 kB после build.

### Улучшены ответы AI-агента
- Ответы больше не начинаются с грубого `ОТВЕТ`.
- Добавлен стабильный формат:
  - `### Краткий вывод`
  - bullets или короткое объяснение
  - `### Источник`
  - `### Ограничения`, если они нужны
- Добавлен post-processing Qwen answer markdown:
  - удаляет shouty labels;
  - добавляет источник, если модель его не указала;
  - ограничивает слишком длинные ответы;
  - нормализует not_found ответы.
- Переписан deterministic map response:
  - нормализует плохие названия карт;
  - объясняет, что реально видно;
  - отдельно показывает легенду, контуры, скважины, категории, подписи;
  - честно указывает ограничения Vision/OCR.
- Добавлена очистка visual title/text, чтобы пользователь не видел внутреннюю фразу классификатора.

### Backend quality
- Версия API обновлена до 0.4.0.
- Health/capabilities теперь возвращают 0.4.0.
- Тесты сохранены и проходят.
- API smoke проверен через живой uvicorn сервер.

## Что ещё осталось для настоящего production

- Подключить полноценный persistent vector backend: Qdrant сейчас всё ещё не является реальным active retrieval backend.
- Добавить E2E tests через Playwright/Cypress.
- Добавить user accounts / roles, если проект будет внешним продуктом.
- Реализовать source viewer endpoint для открытия конкретной страницы PDF.
- Добавить streaming ответов.
- Добавить полноценный audit log действий пользователя.
- Настроить CI/CD pipeline.

## Итог

0.4.0 переводит проект из перегруженного demo-dashboard в более профессиональный, минималистичный и понятный geological RAG workspace. UI стал проще, ответы агента стали читабельнее и честнее, а map answers больше не показывают пользователю внутренние технические формулировки классификатора.
