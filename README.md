# GeoDoc AI

**GeoDoc AI** — это геологический RAG-workspace для анализа технических документов, отчётов разведочных работ, таблиц, карт, схем, разрезов и траекторий скважин. Проект помогает быстро находить ответы в больших PDF/DOCX/Excel/CSV-файлах и связывать текстовые данные с картами, рисунками и источниками.

## Что умеет проект

- Загружает геологические документы: **PDF, DOCX, Excel, CSV, изображения и сканы**.
- Извлекает текст, таблицы, подписи к рисункам, карты и визуальные приложения.
- Делит документ на структурированные артефакты: `text`, `table`, `figure`, `map`.
- Использует RAG-поиск по загруженным материалам.
- Отвечает на вопросы по документу простым профессиональным языком.
- Показывает источники, таблицы, карты и рисунки, которые были использованы при ответе.
- Анализирует карты и схемы через Vision/OCR-слой.
- Поддерживает геологические запросы: площади участков, перспективные горизонты, сейсморазведку, проектные скважины, количественную оценку ресурсов, тектонические схемы и карты.
- Имеет отдельное направление для визуализации траекторий скважин.

## Последние исправления

В этой версии улучшено качество ответов и работа визуальных артефактов:

- убрана автоматическая фраза `Краткий вывод`;
- ответный агент теперь пишет 1–2 понятных абзаца вместо склейки OCR-фрагментов;
- перед генерацией ответа evidence очищается от обрывков, повторов, многоточий, названий файлов и шумных списков приложений;
- в финальный ответ передаются только 3–5 наиболее релевантных источников;
- вопросы вида “где указано?” обрабатываются напрямую: агент сначала называет таблицу, рисунок или страницу;
- вкладка **Карты и рисунки** всегда доступна в карточке ответа;
- если у карты/рисунка есть `metadata.preview_data_url`, frontend обязательно показывает изображение;
- если preview не создан, карточка показывает понятное предупреждение и кнопку **Переиндексировать с Vision/OCR**.

## Основная идея

Обычные чат-боты часто просто пересказывают случайные фрагменты документа. GeoDoc AI должен работать иначе:

1. Пользователь загружает геологический документ.
2. Backend разбирает документ на осмысленные части.
3. RAG-слой находит только релевантные фрагменты.
4. Ответный агент превращает найденные данные в понятный вывод.
5. Интерфейс показывает не только текст ответа, но и подтверждающие таблицы, карты, рисунки и источники.

Главная цель проекта — сделать анализ геологических документов быстрым, понятным и проверяемым.

## Backend

Backend отвечает за загрузку, парсинг, индексацию и обработку вопросов.

Основные задачи backend:

- принять файл от пользователя;
- определить тип документа;
- извлечь текст, таблицы, карты и рисунки;
- создать локальный индекс;
- найти релевантные источники под вопрос;
- сформировать ответ через LLM;
- вернуть ответ, таблицы, карты, рисунки и источники во frontend.

## Frontend

Frontend показывает рабочее пространство GeoDoc AI:

- слева — список загруженных документов;
- сверху — переключение между документами и траекториями;
- в центре — диалог с документом;
- в каждом ответе — вкладки:
  - **Ответ** — понятный текстовый ответ;
  - **Таблицы** — найденные таблицы;
  - **Карты и рисунки** — визуальные материалы из документа;
  - **Источники** — страницы и фрагменты, на которых основан ответ.

## Требования к качеству ответов

GeoDoc AI должен отвечать не как сырой OCR и не как простая склейка фрагментов. Ответ должен быть похож на объяснение специалиста:

- без фразы `Краткий вывод`;
- без обрывков текста;
- без `...` внутри предложения;
- без случайной склейки разных страниц;
- без повторов;
- без длинных цитат, если пользователь не просит цитату;
- с прямым ответом на вопрос в первом предложении;
- с указанием таблицы, рисунка или страницы, если это найдено в документе.

Пример хорошего ответа:

> Количественная оценка перспективных ресурсов указана в таблице 12.1. В документе эта таблица относится к разделу с результатами оценки перспективных углеводородов. Рядом также приведены календарный план работ, объёмы геологоразведочных работ на 2022–2028 годы и основные технико-экономические показатели.

## Карты и рисунки

Во вкладке **Карты и рисунки** должны всегда отображаться визуальные превью. Если изображение найдено, карточка должна показывать саму карту или рисунок, а не пустой блок.

Если Vision/OCR не смог прочитать мелкие подписи, карточка всё равно должна отображаться, но с предупреждением:

> Изображение найдено, но мелкие подписи требуют повторной индексации с включённым Vision/OCR.

## Переменные окружения

Создайте файл `backend/.env` и заполните его своими ключами:

```env
APP_ENV=local
APP_HOST=0.0.0.0
APP_PORT=8001
CORS_ORIGINS=http://localhost:5173,http://localhost:8080

PARSER_ENGINE=hybrid
VECTOR_BACKEND=local
QDRANT_URL=http://localhost:6333
QDRANT_COLLECTION=geodoc_ai

DEEPSEEK_OCR_API_KEY=replace-me
DEEPSEEK_OCR_URL=https://llm.alem.ai/v1/chat/completions
DEEPSEEK_OCR_MODEL=deepseek-ocr
OCR_MAX_PAGES=10

QWEN_BASE_URL=https://llm.alem.ai/v1
QWEN_API_KEY=replace-me
QWEN_MODEL=qwen3.6

ENABLE_VISUAL_ANALYSIS=true
VISION_API_KEY=replace-me
VISION_BASE_URL=https://llm.alem.ai/v1
VISION_MODEL=deepseek-ocr
VISION_MAX_PAGES=8
VISION_RENDER_DPI=150

TOP_K=8
SIMILARITY_THRESHOLD=0.05
RERANK=true
```

Важно: реальные API-ключи нельзя публиковать в GitHub. Добавьте `.env` в `.gitignore`.

## Запуск backend

```bash
cd backend
python -m venv .venv
```

Windows PowerShell:

```bash
.\.venv\Scripts\activate
```

macOS/Linux:

```bash
source .venv/bin/activate
```

Установка зависимостей:

```bash
pip install -r requirements.txt
```

Запуск API:

```bash
uvicorn app.main:app --reload --host 0.0.0.0 --port 8001
```

## Запуск frontend

```bash
cd frontend
npm install
npm run dev
```

Обычно frontend запускается на:

```text
http://localhost:5173
```

Backend API:

```text
http://localhost:8001
```

## Рекомендуемый workflow

1. Запустить backend.
2. Запустить frontend.
3. Загрузить PDF или другой документ.
4. Дождаться завершения индексации.
5. Задать вопрос по документу.
6. Проверить вкладки **Ответ**, **Таблицы**, **Карты и рисунки**, **Источники**.
7. Если карты не читаются, включить Vision/OCR и переиндексировать документ.

## Примеры вопросов

- Какая площадь участка недр Каргалы указана в документе?
- Почему палеозойские терригенно-карбонатные отложения считаются перспективными?
- Чем отличаются независимая и зависимая проектные скважины?
- Где указана количественная оценка перспективных ресурсов?
- Где находится тектоническая схема восточного и юго-восточного обрамления?
- Сравни Петропавловскую и Александровскую структуры по роли в истории разведки участка.

## Модуль траекторий

Модуль **Траектории** предназначен для графического отображения проекций траектории скважины.

Планируемые функции:

- список скважин по проекту;
- карта расположения скважин;
- профиль скважины;
- плановая и вертикальная проекция траектории;
- переключение между скважинами;
- открытие полной карты и полного профиля из отчётности;
- визуальное сравнение проектных и фактических траекторий.

## Статус проекта

GeoDoc AI находится в стадии прототипа. Основной функционал уже покрывает загрузку документов, RAG-ответы, таблицы, источники и визуальные артефакты. Следующий важный этап — улучшение качества ответов и стабильное отображение карт/рисунков.

## Авторская идея
